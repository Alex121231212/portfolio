# lead-hunter

Система мониторинга мессенджер-чатов для B2B-лидогенерации: AI-классификация входящих сообщений, оркестрация диалогов, handoff в CRM.

## Что делает

- Подписывается на целевые публичные чаты, читает поток сообщений
- AI-классификатор определяет лидов по контексту реплики (LLM с короткими промптами для дешевизны)
- Оркестрация исходящих диалогов через пул аккаунтов
- Передача готовых лидов менеджеру: CRM-интеграция или внутренние задачи в UI
- Управление пулом прокси с автоназначением аккаунт ↔ прокси
- Алертинг о критических событиях (`spam_block`, `session_revoked`, аномалии)

## Архитектура

```
Мессенджер-чаты
    │
    ▼
┌──────────────┐   ┌─────────────────┐
│  Мониторинг  │──►│ AI-классификатор│──► leads (is_lead=1)
│   (MTProto)  │   │   (LLM + cache) │
└──────────────┘   └─────────────────┘
                         │
                         ▼
                  ┌──────────────┐   ┌──────────────┐
                  │ outreach_q   │──►│ Пул аккаунтов│
                  │ state machine│   │  (DM лиду)   │
                  └──────────────┘   └──────┬───────┘
                                            │
                                            ▼
                                  ┌──────────────────┐
                                  │   sales-agent    │──► CRM
                                  │ (внешний сервис) │   (handoff)
                                  └──────────────────┘
```

## Стек

- **Runtime:** Node.js 20, Express
- **БД:** better-sqlite3 в WAL-режиме
- **Транспорт мессенджера:** GramJS (нативный MTProto-клиент в процессе, без внешних демонов)
- **LLM:** дешёвая модель для классификации, тяжёлая делегируется в [sales-agent](sales-agent.md)
- **Auth:** JWT, rate-limited login
- **Развёртывание:** Docker, Docker Compose, single host

## Ключевые решения

- **Sales-логика вынесена в отдельный сервис** — lead-hunter отвечает за транспорт и операционные задачи (мониторинг, очереди, прокси, аккаунты), conversational AI делегирован channel-agnostic [sales-agent](sales-agent.md). Подключается через feature-flag `USE_SALES_AGENT=true`.
- **State machine на лидах вместо ad-hoc флагов** — `outreach_q` явно описывает переходы (`new` → `contacted` → `replied` → `handoff` / `dropped`), легко аудитить.
- **MTProto в процессе, а не TDLib-демон** — меньше движущихся частей, проще диагностика, можно держать сотни сессий в одном Node-процессе.
- **Автоназначение прокси при импорте аккаунтов** — не нужно вручную мапить, система сама раскидывает по пулу с лимитом по IP.
- **Прогрессивные лимиты для новых аккаунтов** — изначальный warmup-период с дневным лимитом 1 → 4, чтобы избежать антиспам-флагов.
- **Раздельные FloodWait по методам API** — система знает что лимит `ResolveUsername` отдельный от `SendMessage`, и не убивает все операции из-за бана одного метода.

## Уроки эксплуатации (из `docs/EXPERIENCE.md`)

- ISP-прокси работают для старых сессий, для новых — нужны residential
- Mobile-прокси несовместимы с MTProto-TCP в большинстве случаев (HostUnreachable)
- Массовый `ResolveUsername` приводит к FloodWait на ~14 часов, причём бан распространяется на весь IP
- Безопасный cooldown между диалогами — 30–60 минут, не 10–20
