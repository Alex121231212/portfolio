# Kontent Zavod

Универсальная автоматизированная система производства контента: по заданной нише система анализирует поисковый спрос, формирует план тем, генерирует и публикует контент. Смена ниши делается **только через настройки**, без изменения логики.

## Что делает (MVP1, текущее состояние)

- Управление проектами (нишами): ЦА, регион, базовые ключи, проектные промпты
- Управление компаниями: расписание (количество статей в день/месяц), привязка к проекту
- AI-планировщик строит план на 14 тем (с интентами и датами публикации)
- Генерация SEO-статей с meta title / description
- Автоматическое добавление GEO/AEO-блоков (FAQ-формат) в конец статьи
- Контроль качества: лимиты по длине, фильтр запрещённых фраз
- Полу-автоматическая публикация из админки
- Подробное логирование генераций (модель, успех/ошибка)

## Roadmap

- Авто-публикация в WordPress через API (отключено в MVP1)
- Авто-публикация в Telegram, VK, Яндекс Дзен
- Видео-формат (1 тезис = 1 видео, 20–45 сек, 9:16, TTS, субтитры)
- Анализ конкурентов как дополнительный источник тем
- Анализ уже опубликованного контента проекта

## Стек

- **Backend:** Python, модульная архитектура с явными этапами пайплайна
- **БД:** PostgreSQL
- **LLM:** Cometa API (мульти-провайдер через единый прокси)
- **Источники тем:** API поисковых подсказок (Yandex, далее Google)
- **Storage:** MinIO (под медиа и артефакты, готовится для видео)
- **Развёртывание:** Docker Compose, миграции через alembic-style scripts

## Поддерживаемые ниши

Система работает с любой нишей, где есть поисковый спрос и экспертный/коммерческий контент: сертификация продукции, торги по банкротству, юр. услуги, e-commerce, медицина, образование, БАДы и т.д. Смена ниши = смена конфига и проектных промптов, без правки кода.

## Ключевые решения

### Универсальные сущности вместо специфичных

В коде нет «статьи» — есть `ContentUnit`, у которого есть `формат`. Один контент-юнит может иметь несколько форматов: текст для блога, скрипт для видео, пост для соцсети. Это позволяет менять нишу и формат вывода одним переключением конфига.

### Ниша = конфигурация

Проект описывается набором параметров: ключевые слова, коммерческие/информационные триггеры, тон, юридические ограничения, лимиты, регион, источники, LLM-провайдер, проектные промпты. Никакой ниша-специфичной логики в коде.

### Многоуровневая проверка уникальности

Чтобы при долгой работе в одной нише избежать самоканнибализации:

1. **Уникальность темы** — проверка на смысловом уровне (не по тексту), перед добавлением в очередь
2. **Уникальность интента** — одна формулировка, разные интенты → допустимо
3. **Уникальность контент-юнита** — не было ли уже создано для этой темы + интента + формата

При обнаружении повторов система не молча дропает, а решает: отклонить / понизить приоритет / пометить как производную (обновление, расширение, детализация).

### Логирование LLM-экономики

`GenerationLog` фиксирует каждый LLM-вызов: модель, успех/ошибка, токены, стоимость, версия промпта, провайдер. Можно подбирать оптимум по цене на разных этапах пайплайна.

### GEO/AEO как first-class фича

Контент структурируется под извлечение AI-ассистентами:

- чёткие однозначные ответы на вопросы
- краткие смысловые блоки и определения
- информация извлекается без интерпретации
- минимум «воды» во вводных абзацах

Это даёт zero-click-видимость в Алисе/GigaChat помимо классической SEO-выдачи. GEO-блоки в FAQ-формате генерируются отдельным шагом и добавляются в конец статьи.

### Полу-автомат для MVP1

Первая итерация — полу-автоматическая публикация (preview + кнопка «опубликовать» в админке). Авто-публикация в WordPress / Telegram / VK / Дзен — следующий этап после стабилизации качества генерации.
