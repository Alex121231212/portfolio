# Socionics AI

AI-платформа анализа коммуникативных паттернов на основе теории соционики. Система анализирует диалоги пользователя и его контактов, определяет соционические типы и генерирует рекомендации по взаимодействию.

## Что делает

- **P1 — Self Analysis**: определение собственного соционического типа по совокупности диалогов
- **P2 — Contact Analysis**: определение типа конкретного контакта по диалогу с ним
- **P3 — Pair Analysis**: анализ совместимости двух типов и динамики отношений
- **P4 — Goal Scripts**: рекомендации и сценарии взаимодействия для конкретных целей
- **Multi-LLM**: пользователь сам выбирает провайдера (OpenAI, DeepSeek, Anthropic, DeepInfra) с расчётом стоимости
- **Batch Processing**: умная обработка больших объёмов диалогов батчами с агрегацией

## Стек

- **Backend:** Python 3.11+, FastAPI, SQLAlchemy ORM, Alembic-миграции
- **Frontend:** React + Vite + TypeScript + Tailwind
- **БД:** PostgreSQL 13+
- **Кэш:** Redis
- **LLM:** мульти-провайдер (OpenAI, DeepSeek, Anthropic, DeepInfra) через единый клиент
- **Auth:** JWT
- **Развёртывание:** Docker (раздельные образы backend / frontend)

## Ключевые решения

### Промпты вынесены в файлы и версионируются

Промпты P1–P4 живут в `prompts/` как отдельные файлы (`p1_self_type.*`, `p2_contact_type.*` и т.д.). Это позволяет редактировать без перекомпиляции и держать историю изменений в git отдельно от кода.

### Батчинг с агрегацией для P1

P1 анализирует все диалоги пользователя сразу, что не помещается в один контекст. Решение:

1. Фильтрация диалогов с ≥50 сообщениями (отсекаем шум)
2. Группировка по 5 диалогов в батч (max 10 батчей = 50 диалогов)
3. Каждый батч → отдельный LLM-вызов с промптом P1
4. Результаты агрегируются в финальный тип
5. Кэширование результата в БД

Это даёт детерминированную оценку с любого объёма входных данных.

### Учёт зависимостей между анализами

P2 учитывает результат P1: при анализе типа контакта учитывается тип самого пользователя — это улучшает точность, потому что наша интерпретация поведения собеседника зависит от того, кто наблюдатель.

### Cost-aware UX

Перед запуском анализа фронт показывает оценку стоимости в зависимости от выбранного провайдера и модели. Пользователь видит trade-off (точность vs цена) и осознанно выбирает.

### Кэширование результатов на уровне БД

Один и тот же диалог не пере-анализируется. Если контакт не написал новых сообщений с момента предыдущего P2 — отдаём кэшированный результат. При новых сообщениях — инкрементальный пересчёт.

## Безопасность

- JWT-авторизация
- CORS-конфигурация фронта
- Валидация всего ввода (zod-аналог через pydantic)
- API-ключи LLM-провайдеров в `.env`, не в БД
