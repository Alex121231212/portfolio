# Корпоративный AI-ассистент команды

> Внутренний продукт, описание архитектурное. Конкретный заказчик, выбор моделей и используемые внешние сервисы — под NDA.

Система для автоматизации работы с записями встреч и поддержания «живого» документа состояния команды. AI-ассистент команды отвечает на вопросы, генерирует артефакты и держит в голове весь контекст происходящего.

## Что делает

- Принимает записи встреч (аудио/видео любого формата)
- Транскрибирует с разделением спикеров
- Сопоставляет голоса с эталонами участников команды
- Формирует структурированное саммари
- Автоматически обновляет «живой документ состояния команды»
- Отвечает на вопросы команды через корпоративный мессенджер с доступом к истории встреч и состоянию проектов
- Генерирует артефакты по запросу: код, документы, спецификации, инструкции

## Стек

- **Backend:** Python, FastAPI, асинхронный пайплайн обработки
- **Хранение:** PostgreSQL для транскриптов, саммари, артефактов и состояния команды
- **Транскрипция:** через корпоративную LLM-инфраструктуру (не уточняется)
- **Диаризация:** библиотека спикер-диаризации с voice matching
- **Агент:** корпоративная LLM с нативным tool-calling
- **Канал доставки:** корпоративный мессенджер (адаптер скрыт от ядра)

## Ключевые решения

### Два уровня LLM с разной экономикой

В системе используются модели разного уровня для разных задач:

- **Дешёвая модель** — массовая работа с транскриптами, обновление документа состояния, фильтрация. Низкая стоимость на токен, объём вызовов высокий.
- **Сильная модель** — ядро AI-ассистента, где важна глубина рассуждений, длинный контекст и точное следование инструкциям. Цена выше, объём вызовов низкий.

Это даёт оптимальное соотношение цены/качества: рутина крутится дёшево, важные разговоры — на топ-модели.

### «Живой документ состояния» вместо логов встреч

Ключевая концепция: вместо того чтобы хранить только саммари каждой встречи отдельно, система держит **один документ**, описывающий *текущее* состояние команды:

- статусы проектов (что в работе, что заблокировано)
- кто чем занят прямо сейчас
- открытые задачи с дедлайнами
- актуальные риски и блокеры
- недавние ключевые решения

Документ **перезаписывается целиком** после каждой встречи: `текущий документ + новое саммари → LLM → новый документ`. Это даёт ассистенту мгновенный «срез» состояния без необходимости каждый раз поднимать всю историю.

### Tool-use вместо агентского фреймворка

Рассматривались варианты:

1. Простой промпт без инструментов — мало возможностей подтянуть данные по запросу
2. **LLM API + native tool_use** — LLM сама решает когда вызвать инструмент ✓
3. Полноценный agent SDK — оверкилл для чат-ассистента с доступом к данным

Выбран вариант 2. Без фреймворков — просто описание функций в API + цикл обработки. Максимум 5 tool-вызовов на сообщение (защита от зацикливания).

### Трёхслойный контекст агента

System prompt при каждом сообщении собирается из трёх слоёв:

1. **Статический контекст команды** — описание команды, людей, ролей, стека, процессов. Заполняется один раз вручную, обновляется редко.
2. **Живой документ состояния** — всегда отражает актуальное положение дел.
3. **Последние N саммари встреч** — недавняя оперативная история.

Это даёт ассистенту полное понимание происходящего без необходимости каждый раз подтягивать всё через tool_use — он подтягивает детали только когда они нужны.

### Channel-agnostic дизайн

Агент не знает откуда пришёл вопрос. Канал (мессенджер) — это адаптер с минимальным API, ядро агента работает с абстрактными `conversation_id` и `message`. При необходимости легко добавить новый канал доставки без переписывания логики.
